# MathForge Evaluation Suite — Demonstration Report

**Snapshot:** 2026-07-12  
**Purpose:** Explain what the project can currently measure, what has actually been measured, and what remains before the central behavior claim is proven.

## Executive verdict

MathForge has a credible **evaluation foundation**, but not yet a completed end-to-end evaluation of the trained generator.

- **Implemented:** a frozen, leakage-controlled corpus of 240 held-out competition problems; deterministic export checks; a quality ranker; blind base-versus-adapter comparison code; and teacher-candidate verification workflows.
- **Measured:** one 30-prompt paired generation benchmark on the 7B adapter. It is diagnostic, not a win: the tuned model and base model were effectively tied, while only 7% of tuned outputs were judged well-posed.
- **Not yet measured:** the trained model on the 240-item frozen corpus, or a versioned behavior suite covering crux faithfulness, unique integer answer, no step-stacking, target difficulty, elegance, and novelty.

> **Demo-safe headline:** MathForge has a tamper-checked 240-problem holdout and leakage-controlled training exports. Its first blind 30-pair adapter-versus-base benchmark was diagnostic rather than victorious: 12–13–5 pairwise, with validity as the clear bottleneck. That result motivated the verification-first and combination-generator V2 work.

This distinction matters: verifying the holdout and ranking stored problems proves that the **data infrastructure** works. It does not prove that the trained generator works.

## 1. Target behavior to evaluate

The Brainlift defines the model’s intended behavior as:

> Given a crux insight, generate an original, well-posed AIME-style problem with a unique integer answer whose solution genuinely pivots on that insight, without manufacturing difficulty through step-stacking.

The intended evaluation dimensions are therefore:

1. **Format adherence:** self-contained AIME-style statement and integer answer in `[0, 999]`.
2. **Mathematical validity:** one well-defined interpretation, a unique answer, and a correct solution.
3. **Crux faithfulness:** the supplied insight and required techniques are load-bearing.
4. **Anti-step-stacking:** difficulty comes from an idea rather than many routine operations.
5. **Difficulty targeting:** the output falls in the requested behavioral difficulty band.
6. **Elegance:** low crux count, economy after the insight, and a non-routine compression step.
7. **Novelty:** no copied or near-copied training/evaluation statement or trivial template substitution.
8. **Consistency/robustness:** the behavior survives repeated seeds and awkward but valid requests.

The current benchmark measures only a subset of these dimensions, and it does not condition on held-out cruxes. See [SLM Brainlift](../SLM%20Brainlift.md), especially lines 13, 22, and 34.

### Leakage protection

The main safeguards are real and testable:

- Held-out rows are stored as `split=eval` and `frozen=true`.
- Training selection defaults to TRAIN and non-frozen rows in `src/mathforge/db.py`.
- The exporter removes exact and canonical overlap with evaluation statements in `src/mathforge/integrity.py:176-238`.
- The freeze refuses silent overwrite unless explicitly forced in `src/mathforge/holdout.py:164-180`.
- The manifest hash covers sorted `(problem_id, statement_hash, answer)` triples in `src/mathforge/holdout.py:257-264`.
- Focused holdout and harness tests currently pass (`16 passed` in the 2026-07-12 audit).

The verification command currently passes:

```bash
PYTHONPATH=src uv run mathforge eval-verify
```

### Important readiness boundary

The corpus manifest is **complete and internally verified**, but it is not yet a ready-to-run generator benchmark:

- No preserved run applies the trained SLM to these 240 items.
- The target behavior needs crux-conditioned prompts, but the manifest stores reference problems rather than a versioned bank of evaluation prompts and rubrics.
- All 240 rows have answers. The 150 Omni-MATH items have stronger stored solution/rating coverage; the 90 AIME items do not currently have the same local `Solution`/`Evaluation` coverage.

The verifier also has a narrow scope. It recomputes the hash from the manifest and checks each manifest row’s DB existence, frozen/split status, and statement hash (`src/mathforge/holdout.py:275-303`). It does not currently compare every DB answer/metadata field, reject extra evaluation rows, or anchor the manifest hash externally. “Tamper-checked statement membership” is therefore more exact than “cryptographically immutable database.”

## 4. Measured model comparison: 30 paired generations

### Setup

The implemented notebook cell is [notebooks/bench30_blind_cell.txt](../notebooks/bench30_blind_cell.txt). It compares:

- **Base:** `Qwen/Qwen2.5-Math-7B-Instruct`
- **Tuned:** the same base weights with the historical MathForge 7B LoRA adapter enabled
- **Prompts:** 30 matched requests over Number Theory, Combinatorics, Algebra, Geometry, and Precalculus
- **Sampling:** temperature `1.0`, top-p `0.95`, no fixed random seed, and a prompt nonce
- **Judge:** Claude Opus 4.8
- **Blinding:** source labels hidden and A/B order randomized for the pairwise judgment

### Preserved aggregate result

Source: [Results/bench30_results.tex](../Results/bench30_results.tex).

| Metric | Tuned SLM | Base model | Direction |
|---|---:|---:|---|
| Well-posed rate | 0.07 | 0.10 | Worse |
| Problem elegance (0–5) | 0.82 | 0.65 | Slightly higher |
| Difficulty (1–10) | 2.75 | 1.83 | Higher |

Pairwise judgment:

| Tuned wins | Base wins | Ties |
|---:|---:|---:|
| 12 | 13 | 5 |

Excluding ties, the tuned win fraction is `12/25 = 48%`. A Wilson 95% interval is approximately 30.0%–66.5%, and a two-sided paired sign test gives `p = 1.0`. This is a dead heat, not evidence of superiority.

### What the result says

In this run, tuned outputs received **higher mean difficulty and problem-elegance scores**. They did not clear the main gate: approximately 90% of both models’ generations failed the judge’s well-posedness check, and the tuned pass rate was lower.

The appropriate diagnosis is:

> The benchmark is consistent with harder-looking, somewhat more elegant tuned outputs, but it does not show the reliable mathematical validity required by the behavior spec.

That makes validity—not raw creativity—the binding constraint on the next dataset iteration.

### Reproducibility and judge limitations

The result is useful, but it has material limitations:

- It compares the adapter with its **raw 7B base**, not with a frontier or strong general-purpose generator.
- Thirty pairs provide low statistical power.
- Generation and randomized A/B order were not seeded; there is no immutable run manifest.
- All judgments use one judge family, with no replicated judgments or preserved human calibration study.
- The TeX artifact says “statements only,” while the code sends the full generated text to the judge.
- The generation prompt requests a statement and key idea, yet the judge also asks for answer correctness and solution elegance.
- `Results/mathforge_bench30.txt` ends with `RESULTS: []`; the raw judge outputs needed to reproduce the TeX aggregates are not preserved there.
- `bench30_resultsv2.tex` and `bench30_fullv2.tex` are byte-identical to their non-V2 counterparts, so they are duplicate filenames rather than an independent replication.

The earlier 5-prompt comparison should not be used as headline evidence: its artifacts conflict and the sample is too small.

## 5. Other evaluation evidence

### Best-of-N statement-validity pilot

The prototype in [notebooks/best_of_n_validity_cell.txt](../notebooks/best_of_n_validity_cell.txt) generated 48 candidates and retained 7 after an Opus statement-level well-posed/nontrivial gate: `7/48 = 15%`.

This demonstrates that filtering can improve candidate selection. It does **not** demonstrate full mathematical validity: the gate did not independently verify the final answer and full solution, and several retained examples in [Results/kept_problems.txt](../Results/kept_problems.txt) still have faulty solutions.

### Frontier teacher-candidate verification

The persisted [data/frontier_verification_report.json](../data/frontier_verification_report.json) reports:

| Candidate family | Total | QA persisted | Verified | Human accepted | Human rejected |
|---|---:|---:|---:|---:|---:|
| Combination candidates | 12 | 12 | 12 | 11 | 1 |

This is evidence about **teacher-generated training candidates**, not SLM inference. The 11 historical records used Opus 4.8 as solver and judge; the accepted V2 canary used three Sonnet 4.6 solver attempts and an Opus 4.8 judge. Its generator was also Opus. This improves model separation but remains within the Claude family, so correlated errors are still possible and human adjudication remains essential.

The live database can advance beyond this static artifact as generation jobs finish. Regenerate the report before citing its counts in a recorded demo.

### Stored-problem ranker

`src/mathforge/evalharness.py` implements an elegance-first lexicographic ranker with difficulty, crux economy, and a routine-step penalty behind a correctness gate. It is useful for curation, but two caveats matter:

- Trusted contest sources can bypass the synthetic verification requirement.
- Missing agent well-posedness QA defaults to acceptance.

Thus “keeper” counts from `mathforge rank` are curation counts, not measured model-validity rates.

```bash

```

### E. Present the measured benchmark honestly

Open `Results/bench30_results.tex` or its rendered output and say:

1. “This is 30 matched generations, not the 240-item holdout.”
2. “The tuned outputs received higher judged difficulty/elegance means in this run.”
3. “The pairwise result is a statistical dead heat: 12 tuned, 13 base, and 5 ties; well-posedness is still poor.”
4. “That diagnosis drove the verification-first V2 pipeline.”

Do **not** run `freeze-eval` in a demonstration; freezing is a one-time write operation.

## 8. Next evaluation milestone

The next defensible experiment is a versioned **behavior suite**, not another informal sample gallery.

1. Create a fixed prompt manifest containing held-out cruxes, required technique sets, target difficulty, and adversarial variants.
2. Run at least the base model, historical 7B adapter, and next retrained adapter on identical prompts and multiple recorded seeds.
3. Preserve every prompt, generation setting, model/adapter identifier, raw output, solver result, judge result, and code/data commit.
4. Apply gates in order:
   - parseable AIME format and integer range;
   - independent solution and answer agreement;
   - well-posedness;
   - crux/technique faithfulness;
   - no step-stacking;
   - target difficulty;
   - elegance;
   - novelty against training and frozen corpora.
5. Calibrate model judges on a meaningful blind human-labeled subset and use a different model family from the generator where possible.
6. Report pass rate per gate, all-gates pass rate, paired deltas, confidence intervals, judge disagreement, and an error taxonomy.

The primary success metric should be **all-gates behavior-spec adherence**, with well-posed-and-correct rate reported separately as a hard prerequisite. A frontier comparison becomes meaningful only after the tuned model shows a credible, repeatable improvement over its own base.
